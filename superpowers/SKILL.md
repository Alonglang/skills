# 超能力 — OpenClaw 版

改编自 [obra/superpowers](https://github.com/obra/superpowers)。这是强制性工作流程 — 不是建议。

## 流水线

```
想法 → 头脑风暴 → 规划 → 子代理驱动构建 (TDD) → 代码审查 → 完成分支
```

每个编码任务都遵循此流水线。"太简单不需要设计"永远是错的。

---

## 第一阶段：头脑风暴

**触发条件：** 用户想要构建某个东西。触碰任何代码前激活。

**详见：** [references/brainstorming.md](references/brainstorming.md)

**总结：**
1. 探索项目上下文（文件、文档、最近提交）
2. 提出澄清问题 — **一次一个**，偏好选择题
3. 提出 2-3 种方法，说明权衡 + 推荐方案
4. 分部分展示设计，每部分后获得批准
5. 编写设计文档 → `docs/plans/YYYY-MM-DD-<topic>-design.md` → 提交
6. 移交到 **第二阶段：编写计划**

**硬门槛：** 在用户批准设计之前**不要**写任何代码。

---

## 第二阶段：编写计划

**触发条件：** 设计已批准。由头脑风暴阶段激活。

**详见：** [references/writing-plans.md](references/writing-plans.md)

**总结：**
- 编写逐任务详细实现计划
- 每个任务 = 2-5 分钟：编写测试 → 观看失败 → 实现 → 观看通过 → 提交
- 保存到 `docs/plans/YYYY-MM-DD-<feature>.md`
- 声明：`"我正在使用 writing-plans 技能创建实施计划。"`
- 保存后，提供两种执行模式：
  - **子代理驱动（当前会话）：** 每个任务使用 `sessions_spawn` + 两阶段审查
  - **手动执行：** 用户自己运行任务

---

## 第三阶段：子代理驱动开发

**触发条件：** 计划已存在，用户选择子代理驱动执行。

**详见：** [references/subagent-development.md](references/subagent-development.md)

**每个任务循环（OpenClaw）：
1. `sessions_spawn` 启动一个实现子代理，附带任务 + 完整计划上下文
2. 等待完成宣布
3. `sessions_spawn` 启动规范审查子代理 → 必须确认代码匹配规范
4. `sessions_spawn` 启动代码质量审查子代理 → 必须批准质量
5. 修复任何问题，需要时重新审查
6. 标记任务完成，进入下一个
7. 最后：调度整体代码审查 → 移交到第五阶段

**每个任务都必须遵循TDD。** 详见 [references/tdd.md](references/tdd.md)。

---

## 第四阶段：系统化调试

**触发条件：** 错误、测试失败、意外行为 — 任何技术问题。

**详见：** [references/systematic-debugging.md](references/systematic-debugging.md)

**硬门槛：** 不先调查根本原因就不修复。

**四个阶段：**
1. 根本原因调查（读取错误、复现、检查最近变更、追踪数据流）
2. 模式分析（找到可用示例、比较、识别差异）
3. 假设 + 测试（一次一个假设，测试证明/证伪）
4. 修复 + 验证（在根源修复而非症状；验证修复不破坏任何东西）

---

## 第五阶段：完成分支

**触发条件：** 所有任务完成，所有测试通过。

**详见：** [references/finishing-branch.md](references/finishing-branch.md)

**总结：**
1. 验证所有测试通过
2. 确定基础分支
3. 展示 4 个选项：本地合并 / 推送 + PR / 保留 / 丢弃
4. 执行选择
5. 清理

---

## OpenClaw 子代理调度模式

调度实现者或审查者子代理时，使用 `sessions_spawn`：

```
目标：[一句话]
上下文：[为什么重要，哪个计划文件]
文件：[精确路径]
约束：[什么不能做 — 不要范围蔓延，仅TDD
验证：[如何确认成功 — 测试通过，特定命令]
任务文本：[从计划粘贴完整任务]
```

运行 `sessions_spawn`，将任务作为详细提示。子代理会自动宣布结果。

## 核心原则

- **一次一个问题** 在头脑风暴期间
- **始终TDD** — 先写失败测试，删除测试前写的代码
- **YAGNI** — 从所有设计中移除不必要功能
- **DRY** — 不要重复
- **系统化优先于即兴** — 尤其在时间压力下遵循流程
- **证据优先于宣称** — 验证后才宣布成功
- **频繁提交** — 每次绿色测试后提交
