# Skill Creator - 技能创建器

一个用于创建新技能并迭代改进它们的技能。

从高层次来看，创建技能的过程如下：

- 确定你希望技能做什么以及大致应该怎么做
- 编写技能草稿
- 创建几个测试提示词，并在这些提示词上运行有权访问该技能的 Claude
- 帮助用户定性和定量地评估结果
  - 在运行进行的后台，起草一些定量评估（如果还没有评估的话；如果有，你可以直接使用或根据需要进行修改）。然后向用户解释它们（如果它们已经存在，则解释现有的评估）
  - 使用 `eval-viewer/generate_review.py` 脚本向用户展示结果以供查看，并让他们查看定量指标
- 根据用户对结果评估的反馈重写技能（以及如果从定量基准测试中发现了任何明显的缺陷）
- 重复直到满意
- 扩展测试集并再次尝试更大规模的测试

使用该技能时，你的工作是确定用户处于这个过程的哪个阶段，然后介入并帮助他们推进这些阶段。例如，也许他们会说"我想为 X 制作一个技能"。你可以帮助缩小他们的意思、编写草稿、编写测试用例、确定他们想要如何评估、运行所有提示词，然后重复。

另一方面，也许他们已经有了技能草稿。在这种情况下，你可以直接进入循环的评估/迭代部分。

当然，你应该始终灵活，如果用户像"我不需要运行一堆评估，就让我们一起交流"，你可以这样做。

然后在技能完成后（但同样，顺序是灵活的），你也可以运行技能描述优化器，我们有一个单独的脚本来优化技能的触发。

很酷？很酷。

## 与用户沟通

技能创建器可能会被各种熟悉代码术语的人使用。如果你没听说过（你怎么可能知道，这只是最近才开始的事情），现在有一种趋势，Claude 的力量正在激励水管工打开终端、父母和祖父母谷歌"如何安装 npm"。另一方面，大多数用户可能相当精通计算机。

所以请注意上下文线索，以了解如何组织你的沟通！在默认情况下，只是为了给你一些想法：

- "evaluation"（评估）和"benchmark"（基准测试）是边缘的，但还可以
- 对于"JSON"和"assertion"（断言），你想在用户明确知道它们是什么之前，在不解释它们的情况下使用它们

如果你不确定，可以简要解释术语，如果你不确定用户是否能理解，可以用简短的定义来澄清术语。

## 创建技能

### 捕获意图

首先了解用户的意图。当前对话可能已经包含用户想要捕获的工作流程（例如，他们说"把它变成一个技能"）。如果是这样，首先从对话历史中提取答案——使用的工具、步骤顺序、用户所做的更正、观察到的输入/输出格式。用户可能需要填补空白，并在进行下一步之前进行确认。

1. 这个技能应该让 Claude 能够做什么？
2. 什么时候应该触发这个技能？（什么用户短语/上下文）
3. 预期的输出格式是什么？
4. 我们是否应该设置测试用例来验证技能是否有效？具有客观可验证输出的技能（文件转换、数据提取、代码生成、固定工作流程步骤）从测试用例中受益。具有主观输出的技能（写作风格、艺术）通常不需要它们。根据技能类型建议适当的默认值，但让用户决定。

### 采访和研究

主动询问边缘情况、输入/输出格式、示例文件、成功标准和依赖关系。等待起草测试提示词，直到你把这部分搞清楚。

检查可用的 MCP——如果对研究有用（搜索文档、查找类似的技能、查找最佳实践），如果有可用的子代理，请通过子代理并行研究，否则直接进行。准备上下文以减少用户的负担。

### 编写 SKILL.md

根据用户访谈，填写这些组件：

- **name**：技能标识符
- **description**：何时触发，它做什么。这是主要的触发机制——包括技能做什么和何时使用它的特定上下文。所有"何时使用"的信息都在这里，不在正文中。注意：目前 Claude 倾向于"欠触发"技能——在它们有用时不使用它们。为了对抗这种情况，请使技能描述稍微"强势"一点。因此，例如，不要写"如何构建一个简单的快速仪表板来显示内部 Anthropic 数据"，你可能会写"如何构建一个简单的快速仪表板来显示内部 Anthropic 数据。确保在用户提到仪表板、数据可视化、内部指标或想要显示任何类型的公司数据时使用此技能，即使他们没有明确要求'仪表板'。"
- **compatibility**：所需的工具、依赖项（可选，很少需要）
- **技能的其余部分 :)**

### 技能编写指南

#### 技能的解剖结构

```
skill-name/
├── SKILL.md (必需)
│   ├── YAML frontmatter (name, description 必需)
│   └── Markdown 指令
└── 捆绑资源 (可选)
    ├── scripts/    - 用于确定性/重复性任务的可执行代码
    ├── references/ - 根据需要加载到上下文中的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

#### 渐进式披露

技能使用三级加载系统：
1. **元数据**（name + description）- 始终在上下文中（约 100 个单词）
2. **SKILL.md 正文** - 每当技能触发时都在上下文中（理想情况下少于 500 行）
3. **捆绑资源** - 按需（无限，脚本可以在不加载的情况下执行）

这些字数是近似的，如果需要，你可以随意延长时间。

**关键模式：**
- 保持 SKILL.md 在 500 行以下；如果你接近此限制，请添加额外的层次结构以及清晰的指针，指示使用该技能的模型接下来应该去哪里进行跟进。
- 从 SKILL.md 中清楚地引用文件，并提供关于何时阅读它们的指导
- 对于大型参考文件（>300 行），包括目录

**域组织**：当技能支持多个域/框架时，按变体组织：
```
cloud-deploy/
├── SKILL.md (工作流程 + 选择)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claude 只读取相关的参考文件。

#### 无意外原则

不用说，技能不得包含恶意软件、利用代码或任何可能损害系统安全的内容。如果描述了技能的内容，它不应该让用户感到惊讶。不要创建误导性技能或旨在促进未经授权访问、数据泄露或其他恶意活动的技能的请求。像"作为 XYZ 扮演角色"之类的东西是可以的。

#### 编写模式

倾向于在指令中使用祈使形式。

**定义输出格式** - 你可以这样做：
```markdown
## 报告结构
始终使用此确切模板：
# [标题]
## 执行摘要
## 主要发现
## 建议
```

**示例模式** - 包含示例很有用。你可以这样格式化它们（但如果示例中包含"输入"和"输出"，你可能需要稍微偏离）：
```markdown
## 提交消息格式
**示例 1：**
输入：使用 JWT 令牌添加用户身份验证
输出：feat(auth): 实现基于 JWT 的身份验证
```

### 编写风格

尝试向模型解释为什么事情很重要，而不是用强硬的陈旧的 MUST。使用心智理论，并尝试使技能通用，而不是对特定示例过于狭窄。首先编写草稿，然后以新的眼光看待它并进行改进。

### 测试用例

编写技能草稿后，提出 2-3 个现实的测试提示词——真正用户实际会说的话。与用户分享它们：[你不必使用这种确切的语言]"这里有一些我想尝试的测试用例。这些看起来对吗，还是你想添加更多？"然后运行它们。

将测试用例保存到 `evals/evals.json`。暂不编写断言——只是提示词。你将在运行进行时在下一步中起草断言。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务提示词",
      "expected_output": "预期结果的描述",
      "files": []
    }
  ]
}
```

有关完整架构（包括你稍后将添加的 `assertions` 字段），请参阅 `references/schemas.md`。

## 运行和评估测试用例

本部分是一个连续的序列——不要在中间停止。不要使用 `/skill-test` 或任何其他测试技能。

将结果放在 `<skill-name>-workspace/` 中，作为技能目录的兄弟目录。在工作区中，按迭代组织结果（`iteration-1/`、`iteration-2/` 等），在其中，每个测试用例都有一个目录（`eval-0/`、`eval-1/` 等）。不要预先创建所有这些——只需在进行时创建目录。

### 步骤 1：在同一回合中生成所有运行（带技能 和基线）

对于每个测试用例，在同一回合中生成两个子代理——一个有技能，一个没有。这很重要：不要先生成带技能的运行，然后再回来进行基线检查。一次性启动所有内容，以便它们几乎同时完成。

**带技能的运行：**

```
执行此任务：
- 技能路径：<path-to-skill>
- 任务：<eval prompt>
- 输入文件：<eval files 如果有的话，或 "none">
- 将输出保存到：<workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 要保存的输出：<用户关心的内容——例如 ".docx 文件"、"最终的 CSV">
```

**基线运行**（相同的提示词，但基线取决于上下文）：
- **创建新技能**：根本没有技能。相同的提示词，无技能路径，保存到 `without_skill/outputs/`。
- **改进现有技能**：旧版本。在编辑之前，对技能进行快照（`cp -r <skill-path> <workspace>/skill-snapshot/`），然后将基线子代理指向快照。保存到 `old_skill/outputs/`。

为每个测试用例编写一个 `eval_metadata.json`（断言现在可以为空）。为每个评估提供一个基于其测试内容的描述性名称——而不仅仅是"eval-0"。也使用此名称作为目录名称。如果此迭代使用新的或修改的评估提示词，请为每个新评估目录创建这些文件——不要假设它们从以前的迭代中继承。

```json
{
  "eval_id": 0,
  "eval_name": "描述性名称在这里",
  "prompt": "用户的任务提示词",
  "assertions": []
}
```

### 步骤 2：在运行进行时，起草断言

不要只是等待运行完成——你可以富有成效地使用这段时间。为每个测试用例起草定量断言并向用户解释它们。如果 `evals/evals.json` 中已存在断言，请审查它们并解释它们检查的内容。

好的断言是客观可验证的，并且具有描述性名称——它们应该在基准测试查看器中清楚地阅读，以便浏览结果的人立即理解每个断言检查的内容。主观技能（写作风格、设计质量）最好进行定性评估——不要将断言强加给需要人类判断的事情。

起草后，使用断言更新 `eval_metadata.json` 文件和 `evals/evals.json`。还要向用户解释他们将在查看器中看到的内容——定性输出和定量基准测试。

### 步骤 3：当运行完成时，捕获计时数据

当每个子代理任务完成时，你会收到一个包含 `total_tokens` 和 `duration_ms` 的通知。立即将此数据保存到运行目录中的 `timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是捕获此数据的唯一机会——它通过任务通知传递，并且不会在其他地方持久化。在到达时处理每个通知，而不是尝试批量处理它们。

### 步骤 4：评分、聚合和启动查看器

所有运行完成后：

1. **对每个运行进行评分**——生成一个评分器子代理（或内联评分）读取 `agents/grader.md` 并根据输出评估每个断言。将结果保存到每个运行目录中的 `grading.json`。grading.json 期望数组必须使用字段 `text`、`passed` 和 `evidence`（而不是 `name`/`met`/`details` 或其他变体）——查看器取决于这些确切的字段名称。对于可以通过编程方式检查的断言，编写并运行脚本而不是用眼睛盯着它——脚本更快、更可靠，并且可以在迭代之间重用。

2. **聚合到基准测试中**——从 skill-creator 目录运行聚合脚本：
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   这会产生 `benchmark.json` 和 `benchmark.md`，其中包含每个配置的 pass_rate、时间和标记，包括平均值 ± 标准偏差和差值。如果手动生成 benchmark.json，请参阅 `references/schemas.md` 了解查看器期望的确切架构。
将每个 with_skill 版本放在其基线对应项之前。

3. **进行分析师分析**——读取基准测试数据并发现聚合统计信息可能隐藏的模式。请参阅 `agents/analyzer.md`（"分析基准测试结果"部分）了解要查找的内容——比如无论技能如何总是通过的断言（无判别力）、高方差评估（可能不稳定）以及时间/标记权衡。

4. **启动查看器**，其中包含定性输出和定量数据：
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   对于迭代 2+，还要传递 `--previous-workspace <workspace>/iteration-<N-1>`。

   **Cowork / 无头环境：**如果 `webbrowser.open()` 不可用或环境没有显示，请使用 `--static <output_path>` 编写独立的 HTML 文件，而不是启动服务器。当用户点击"提交所有评论"时，反馈将下载为 `feedback.json` 文件。下载后，将 `feedback.json` 复制到下一次迭代的工作区目录中进行拾取。

注意：请使用 generate_review.py 创建查看器；无需编写自定义 HTML。

5. **告诉用户**一些类似这样的话："我已经在你的浏览器中打开了结果。有两个选项卡——'输出'让你点击每个测试用例并留下反馈，'基准测试'显示定量比较。完成后，回到这里告诉我。"

### 用户在查看器中看到的内容

"输出"选项卡一次显示一个测试用例：
- **提示词**：给出的任务
- **输出**：技能生成的文件，在可能的情况下内联渲染
- **之前的输出**（迭代 2+）：显示上次迭代的输出的折叠部分
- **正式等级**（如果运行了评分）：显示断言通过/失败的折叠部分
- **反馈**：在他们键入时自动保存的文本框
- **之前的反馈**（迭代 2+）：他们上次的评论，显示在文本框下方

"基准测试"选项卡显示统计摘要：每个配置的通过率、时间和标记使用情况，包括每个评估的细目和分析师观察结果。

导航通过上一个/下一个按钮或箭头键。完成后，他们点击"提交所有评论"，这将所有反馈保存到 `feedback.json`。

### 步骤 5：阅读反馈

当用户告诉你他们完成时，读取 `feedback.json`：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "图表缺少轴标签", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完美，喜欢这个", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空反馈意味着用户认为它很好。专注于改进用户有具体投诉的测试用例。

完成后终止查看器服务器：

```bash
kill $VIEWER_PID 2>/dev/null
```

## 改进技能

这是循环的核心。你已经运行了测试用例，用户审查了结果，现在你需要根据他们的反馈使技能更好。

### 如何考虑改进

1. **从反馈中归纳。**这里发生的大概情况是，我们试图创建可以数百万次使用的技能（也许字面意思，也许甚至更多谁知道），跨越许多不同的提示词。在这里你和用户只对几个例子一遍又一遍地迭代，因为这有助于加快速度。用户非常了解这些例子，对他们来说评估新输出很快。但是，如果你和用户共同开发的技能只适用于这些例子，它是无用的。与其投入繁琐的过度拟合更改或压抑的约束性 MUST，如果有一些顽固的问题，你可以尝试分支并使用不同的隐喻，或推荐不同的工作模式。尝试相对便宜，也许你会遇到一些很棒的东西。

2. **保持提示词精简。**删除没有发挥其作用的东西。确保阅读转录内容，而不仅仅是最终输出——如果看起来技能使模型浪费大量时间做没有成效的事情，你可以尝试摆脱技能中使其这样做的部分，看看会发生什么。

3. **解释为什么。**努力解释你要求模型做的一切背后的**原因**。今天的 LLM 是*聪明的*。它们有良好的心智状态，当给出良好的束缚时，它们可以超越死板的指令，真正实现事情。即使来自用户的反馈是简短或沮丧的，也要尝试真正理解任务以及为什么用户写他们写的内容，以及他们实际写了什么，然后将这种理解传递到指令中。如果你发现自己在所有大写中写 ALWAYS 或 NEVER，或者使用超级僵化的结构，那就是一个黄牌标志——如果可能的话，重构并解释推理，以便模型理解你要求的事情为什么重要。这是一种更人性化、更强大、更有效的方法。

4. **查找测试用例中的重复工作。**阅读测试运行的转录内容，并注意是否所有子代理都独立地编写了类似的辅助脚本或采用了相同的多步骤方法来处理某些事情。如果所有 3 个测试用例都导致子代理编写 `create_docx.py` 或 `build_chart.py`，那么这是一个强烈的信号，说明技能应该捆绑该脚本。编写一次，将其放在 `scripts/` 中，并告诉技能使用它。这节省了每次未来调用都重新发明轮子。

这项任务相当重要（我们这里试图创造数十亿美元的经济价值！），你的思考时间不是瓶颈；花点时间真正考虑事情。我建议先写一个草稿修订，然后以新的眼光看待它并进行改进。真正尽力进入用户的头脑，了解他们想要和需要什么。

### 迭代循环

改进技能后：

1. 将你的改进应用于技能
2. 将所有测试用例重新运行到新的 `iteration-<N+1>/` 目录中，包括基线运行。如果你正在创建新技能，基线始终是 `without_skill`（没有技能）——这在迭代之间保持不变。如果你正在改进现有技能，请使用你的判断来确定什么作为基线有意义：用户带来的原始版本，还是前一次迭代。
3. 使用 `--previous-workspace` 指向前一次迭代启动审阅器
4. 等待用户审查并告诉你他们完成了
5. 阅读新的反馈，再次改进，重复

继续直到：
- 用户说他们满意
- 反馈都为空（一切看起来都很好）
- 你没有取得有意义的进展

## 高级：盲比较

对于你想要更严格地比较技能的两个版本的情况（例如，用户询问"新版本是否真的更好？"），有一个盲比较系统。请阅读 `agents/comparator.md` 和 `agents/analyzer.md` 了解详细信息。基本思想是：将两个输出提供给独立的代理，而不告诉它哪个是哪个，并让它判断质量。然后分析为什么赢家赢了。

这是可选的，需要子代理，大多数用户不需要它。人工审查循环通常就足够了。

## 描述优化

SKILL.md frontmatter 中的描述字段是决定 Claude 是否调用技能的主要机制。创建或改进技能后，提议优化描述以提高触发准确性。

### 步骤 1：生成触发评估查询

创建 20 个评估查询——混合应该触发和不应该触发的查询。保存为 JSON：

```json
[
  {"query": "用户提示词", "should_trigger": true},
  {"query": "另一个提示词", "should_trigger": false}
]
```

查询必须是现实的，并且是 Claude Code 或 Claude.ai 用户实际会键入的。不是抽象的请求，而是具体的、有明确细节的请求。例如，文件路径、关于用户工作或情况的个人上下文、列名称和值、公司名称、URL。一点点背景故事。有些可能是小写的或包含缩写、拼写错误或随意语言。使用不同长度和组合，并专注于边缘情况而不是使它们清晰明确（用户将有机会签署）。

不好的：`"格式化此数据"`、`"从 PDF 中提取文本"`、`"创建图表"`

好的：`"好吧，所以我老板刚刚给我发了一个 xlsx 文件（在我的下载中，叫做像 'Q4 sales final FINAL v2.xlsx' 之类），她希望我添加一个显示利润率百分比的列。收入在 C 列，成本在 D 列中我想"`

对于 **should_trigger** 查询（8-10 个），考虑覆盖。你需要相同意图的不同措辞——一些正式的，一些随意的。包括用户没有明确命名技能或文件类型但明确需要它的情况。抛出一些不常见的用例，以及此技能与另一个技能竞争但应该获胜的情况。

对于 **should_not_trigger** 查询（8-10 个），最有价值的是接近未命中的查询——与技能共享关键字或概念但实际上需要不同事物的查询。考虑相邻域、幼稚的关键字匹配会触发但不应该的模糊措辞，以及查询触及技能所做的事情但在另一个工具更合适的上下文中的情况。

要避免的关键事情：不要使 should-not-trigger 查询明显无关。作为一个 PDF 技能的负面测试的"编写斐波那契函数"太简单了——它不测试任何东西。负面情况应该真正具有挑战性。

### 步骤 2：与用户一起审查

使用 HTML 模板向用户显示评估集以供审查：

1. 从 `assets/eval_review.html` 读取模板
2. 替换占位符：
   - `__EVAL_DATA_PLACEHOLDER__` → 评估项的 JSON 数组（周围没有引号——它是一个 JS 变量赋值）
   - `__SKILL_NAME_PLACEHOLDER__` → 技能的名称
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 技能的当前描述
3. 写入临时文件（例如，`/tmp/eval_review_<skill-name>.html`）并打开它：`open /tmp/eval_review_<skill-name>.html`
4. 用户可以编辑查询、切换 should-trigger、添加/删除条目，然后点击"导出评估集"
5. 文件下载到 `~/Downloads/eval_set.json`——检查下载文件夹以获取最新版本，如果有多个（例如，`eval_set (1).json`）

这一步很重要——糟糕的评估查询导致糟糕的描述。

### 步骤 3：运行优化循环

告诉用户："这将需要一些时间——我将在后台运行优化循环并定期检查。"

将评估集保存到工作区，然后在后台运行：

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

使用系统提示词中的模型 ID（为当前会话提供动力的那个），以便触发测试与用户的实际体验相匹配。

在它运行时，定期查看输出，以便向用户提供关于它处于哪个迭代状态以及分数看起来如何的更新。

这会自动处理完整的优化循环。它将评估集拆分为 60% 的训练和 40% 的保留测试，评估当前描述（运行每个查询 3 次以获得可靠的触发率），然后使用扩展思考调用 Claude 根据失败的内容提出改进。它在训练和测试上重新评估每个新描述，最多迭代 5 次。完成后，它在浏览器中打开一个 HTML 报告，显示每次迭代的结果，并返回带有 `best_description` 的 JSON——通过测试分数而不是训练分数选择以避免过度拟合。

### 技能触发的工作原理

了解触发机制有助于设计更好的评估查询。技能以其名称+描述出现在 Claude 的 `available_skills` 列表中，Claude 根据该描述决定是否咨询技能。要知道的重要事情是，Claude 只为其不容易自己处理的任务咨询技能——简单的、一步查询，如"阅读此 PDF"，即使描述完全匹配也可能不会触发技能，因为 Claude 可以使用基本工具直接处理它们。当描述匹配时，复杂的、多步骤或专门的查询会可靠地触发技能。

这意味着你的评估查询应该是实质性的，以至于 Claude 实际上会从咨询技能中受益。像"阅读文件 X"这样的简单查询是糟糕的测试用例——无论描述质量如何，它们都不会触发技能。

### 步骤 4：应用结果

从 JSON 输出中获取 `best_description` 并更新技能的 SKILL.md frontmatter。向用户显示之前/之后并报告分数。

### 打包和呈现（仅当 `present_files` 工具可用时）

检查你是否有权访问 `present_files` 工具。如果不，请跳过此步骤。如果有，打包技能并将 .skill 文件呈现给用户：

```bash
python -m scripts.package_skill <path/to/skill-folder>
```

打包后，引导用户到生成的 `.skill` 文件路径，以便他们可以安装它。

## Claude.ai 特定说明

在 Claude.ai 中，核心工作流程是相同的（草稿 → 测试 → 审查 → 改进 → 重复），但由于 Claude.ai 没有子代理，一些机制会发生变化。这里需要适应：

**运行测试用例**：没有子代理意味着没有并行执行。对于每个测试用例，阅读技能的 SKILL.md，然后按照其指令自己完成测试提示词。一个接一个地做。这不如独立的子代理严格（你编写了技能并且你也在运行它，所以你有完整的上下文），但它是一个有用的健全性检查——人工审查步骤补偿了。跳过基线运行——只使用技能按要求完成任务。

**审查结果**：如果你无法打开浏览器（例如，Claude.ai 的 VM 没有显示，或者你在远程服务器上），则完全跳过浏览器审阅器。相反，直接在对话中呈现结果。对于每个测试用例，显示提示词和输出。如果输出是用户需要看到的文件（如 .docx 或 .xlsx），请将其保存到文件系统并告诉他们它在哪里以便他们可以下载和检查。内联询问反馈："这看起来怎么样？你想改变什么？"

**基准测试**：跳过定量基准测试——它依赖于没有子代理就没有意义的基准比较。专注于来自用户的定性反馈。

**迭代循环**：与之前相同——改进技能，重新运行测试用例，询问反馈——只是在中间没有浏览器审阅器。如果你有文件系统，你仍然可以在文件系统上按迭代目录组织结果。

**描述优化**：此部分需要 `claude` CLI 工具（特别是 `claude -p`），它仅在 Claude Code 中可用。如果你在 Claude.ai 上，请跳过它。

**盲比较**：需要子代理。跳过它。

**打包**：`package_skill.py` 脚本可以在任何地方使用 Python 和文件系统工作。在 Claude.ai 上，你可以运行它，用户可以下载生成的 `.skill` 文件。

## Cowork 特定说明

如果你在 Cowork 中，主要要知道的是：

- 你有子代理，所以主要工作流程（并行生成测试用例、运行基线、评分等）都有效。（但是，如果你遇到超时严重问题，可以串行而不是并行运行测试提示词。）
- 你没有浏览器或显示，所以在生成评估查看器时，请使用 `--static <output_path>` 编写独立的 HTML 文件，而不是启动服务器。然后提供一个用户可以点击以在浏览器中打开 HTML 的链接。
- 无论出于什么原因，Cowork 设置似乎使 Claude 不倾向于在运行测试后生成评估查看器，所以只是为了重申：无论你在 Cowork 还是 Claude Code 中，在运行测试后，你应该始终生成评估查看器供人类在修订技能本身并尝试进行更正之前查看示例，使用 `generate_review.py`（而不是编写自己的精品 html 代码）。提前道歉，但我会全大写：在自己评估输入之前*先*生成评估查看器。你想尽快把它们放在人类面前！
- 反馈的工作方式不同：由于没有运行的服务器，查看器的"提交所有评论"按钮会将 `feedback.json` 下载为文件。你可以从那里读取它（你可能需要先请求访问）。
- 打包工作——`package_skill.py` 只需要 Python 和文件系统。
- 描述优化（`run_loop.py` / `run_eval.py`）应该可以在 Cowork 中正常工作，因为它通过子进程使用 `claude -p`，而不是浏览器，但请保存它，直到你完全完成使技能成为并且用户同意它处于良好状态。
